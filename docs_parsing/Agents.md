# Инструкция по парсингу документаций (Агенты)

Данная директория предназначена для автоматизированного сбора (парсинга) официальных документаций различных инструментов и библиотек в удобном Markdown формате. Сбор документаций позволяет формировать локальную базу знаний.

> **🔥 ОБЯЗАТЕЛЬНОЕ ПРАВИЛО:** После каждого успешного создания парсера для новой, ранее не встречавшейся структуры сайта (прорыва), вы **ОБЯЗАНЫ** обновить этот файл `Agents.md`. Необходимо добавить описание новой структуры, алгоритм ее обхода и зафиксировать этот кейс для будущих задач. База знаний должна постоянно эволюционировать!

## Архитектура проекта

Проект инициализирован через `uv` в корне директории `Парсинги документаций/`. Все зависимости хранятся централизованно в `pyproject.toml`.

Для запуска парсеров используется единый CLI интерфейс (`main.py`), а каждый новый сайт добавляется как отдельный модуль (папка с файлом `parser.py`).

**Команды CLI:**

- Посмотреть все доступные парсеры: `uv run python main.py list`
- Запустить конкретный парсер: `uv run python main.py run <имя_модуля>` (например, `opencode`)
- Запустить все парсеры по очереди: `uv run python main.py run all`

## Используемые навыки и инструменты

Для исследования структуры веб-сайтов документаций **ОБЯЗАТЕЛЬНО** использовать навык **`playwright-cli`**. Это мощный инструмент CLI, позволяющий управлять браузером, делать скриншоты и получать текстовые слепки (снэпшоты) DOM-дерева страницы.

### Подробная инструкция по использованию playwright-cli

Инструмент `playwright-cli` установлен глобально и работает через поддержание процессов (сессий) браузера. Вы можете отправлять команды напрямую через `bash` без префикса `npx`.

**1. Запуск сессии и открытие страницы**
```bash
playwright-cli -s=my_session open https://example.com/docs
```
*(Флаг `-s=my_session` привязывает команды к конкретному окну браузера).*

**2. Создание снимка (Snapshot)**
```bash
playwright-cli -s=my_session snapshot
```
В консоли появится ссылка на сохраненный `.yml` файл. Прочитайте этот файл (через инструмент `Read`) — в нем лежит семантическое дерево всей страницы. По нему очень легко понять структуру списков, теги и классы элементов без необходимости дампить сырой HTML.

**3. Взаимодействие со страницей**
В `yml` файле вы увидите идентификаторы (например `[ref=e15]`). Вы можете кликать по ним:
```bash
playwright-cli -s=my_session click e15
```

**4. Выполнение произвольного JS (оценка структуры)**
```bash
playwright-cli -s=my_session run-code "const h1 = document.querySelector('h1'); console.log(h1.parentElement.innerHTML);"
```

**5. Ускорение парсинга (опционально)**
```bash
playwright-cli -s=my_session route "**/*.jpg" --status=404
```

**6. Закрытие сессии**
```bash
playwright-cli -s=my_session close
```

Для самого парсинга используется **Python** с применением пакетов: `playwright`, `beautifulsoup4`, `html2text`.

> **Важно:** При парсинге обязательно задавайте реалистичный `user-agent` (например, Chrome 122 на Windows), чтобы избежать блокировок от Cloudflare и других анти-бот систем.

## Паттерны навигации: как исследовать и собирать ссылки

Если при попытке получить ссылки базовым `querySelectorAll('a')` вы не видите всего меню (или не можете определить вложенность), **остановитесь и проведите исследование структуры** с помощью `playwright-cli` и выполнения JS-кода в консоли. 

На данный момент выявлены следующие паттерны структур и алгоритмы работы с ними:

### Тип 1: Классические вложенные списки (`<ul>` / `<li>`)
- **Кейс:** OpenCode
- **Симптомы:** Меню представляет собой дерево тегов `<ul>`, внутри которых лежат `<li>` со ссылками или другими `<ul>`.
- **Алгоритм:** Используйте рекурсивный обход DOM-дерева (tree walker). Уровень вложенности (depth) увеличивается каждый раз, когда скрипт проваливается в очередной не корневой тег `<ul>`.

### Тип 2: Плоский список с визуальной иерархией (CSS Padding)
- **Кейс:** Google Antigravity
- **Симптомы:** В HTML все элементы меню лежат на одном уровне (например, просто список `<div><a>...</a></div>` подряд). Вложенность существует только визуально благодаря CSS-свойству `padding-left`.
- **Алгоритм:** Соберите все ссылки подряд, но для вычисления `depth` программно читайте inline-стили элемента: `parseFloat(a.style.paddingLeft)`. Если `padding` большой — это подраздел, если маленький — родитель.

### Тип 3: Динамические аккордеоны (Скрытый контент)
- **Кейс:** Payload CMS
- **Симптомы:** При загрузке страницы в DOM присутствуют только заголовки верхнего уровня. Ссылки скрыты или вовсе не отрендерены React/Vue, пока пользователь не раскроет группу. 
- **Алгоритм:** 
  1. Найти все кнопки переключения аккордеонов (через классы или атрибуты вроде `data-state="closed"`).
  2. Программно кликнуть по ним в браузере: `buttons.forEach(b => b.click())`.
  3. Обязательно добавить задержку (`await delay(500)`), чтобы JS-фреймворк сайта успел отрендерить новые ссылки в DOM.
  4. Только после этого собрать ссылки, связывая их с родительскими заголовками.

### Тип 4: Nextra Sidebar с дублированной мобильной/десктопной навигацией
- **Кейс:** Pencil Documentation
- **Симптомы:** В DOM одновременно присутствуют мобильный и десктопный сайдбар. Мобильный `aside` идет первым и может сбивать простой `querySelector('aside')`. Десктопный сайдбар расположен в `aside.nextra-sidebar`, а структура меню выражена как `ul > li`, где группы оформлены через `button[data-href]`, а реальные страницы лежат только в дочерних ссылках `<a>`.
- **Алгоритм:**
  1. Явно брать **десктопный** контейнер `aside.nextra-sidebar`, а не первый попавшийся `aside`.
  2. Обходить первое корневое `ul` рекурсивно по схеме `li -> (a | button) -> ul`.
  3. Использовать `button` только как элементы **групп** оглавления; `data-href` у них может вести на несуществующие index-страницы (`404`), поэтому в итоговый список страниц добавлять только реальные `<a>`.
  4. Основной контент брать из `article > main[data-pagefind-body]` (или ближайшего `main`), удаляя `subheading-anchor`, `iframe` и служебные кнопки.
  5. Блоки кода в Nextra/Shiki приходят как `pre > code.nextra-code` плюс кнопки копирования; перед конвертацией нужно заменить содержимое `pre` на чистый `innerText`, чтобы сохранить переносы строк и не тащить SVG/кнопки в Markdown.

## Валидация и очистка результатов (Решение проблем с версткой)

> **🔥 ПРАВИЛО:** Всегда проверяйте сгенерированный Markdown файл (особенно таблицы и **блоки кода**). Если код разбивается лишними переносами строк или внутри кода появляются странные цифры — модуль должен быть доработан!

### Проблема: Номера строк внутри блоков кода
- **Кейс:** Payload CMS
- **Симптомы:** Современные сайты документаций используют продвинутые библиотеки подсветки синтаксиса (Prism, Shiki). Они разбивают код на `<div class="line">` и добавляют номера строк (например `<span class="lineNumber">1</span>`). При конвертации в Markdown каждый номер строки приклеивается к коду (получается `1 \n\n 2 \n\n const a = 1;`).
- **Решение:** Перед вызовом `html2text` в `page.evaluate` необходимо удалить элементы с номерами строк из DOM. Пример:
  ```javascript
  main.querySelectorAll('[class*="lineNumber"]').forEach(n => n.remove());
  ```
  Также, если код обернут не в стандартный `<pre>`, а в кастомные `<div>`, необходимо вручную извлечь текст и обернуть его в `<pre><code>`:
  ```javascript
  main.querySelectorAll('.code-block-wrap, pre').forEach(block => {
      const text = block.innerText; // Получаем чистый текст без вложенных <span>
      block.innerHTML = '<pre><code>' + text.replace(/</g, '&lt;').replace(/>/g, '&gt;') + '</code></pre>';
  });
  ```

## Как добавить новый сайт для парсинга документаций

### 1. Исследование структуры (через playwright-cli)
- Откройте страницу и снимите `.yml` снапшот.
- Найдите **левое меню навигации** и определите, к какому из паттернов выше оно относится (или откройте новый паттерн).
- Найдите **основной контент страницы** (например, теги `<main>`, `article`, или родительские теги заголовка `h1`), чтобы отсечь лишний мусор.

### 2. Создание структуры модуля
- Создайте новую папку с названием проекта (например, `myproject/`).
- Внутри создайте пустой файл `__init__.py`.
- Создайте скрипт `parser.py`.
- Обязательно зарегистрируйте новый модуль в словаре `PARSERS` в главном файле `main.py`.

### 3. Разработка асинхронного парсера (`parser.py`)
Ваш скрипт должен содержать асинхронную функцию `main()` (или `parse()`) и выполнять следующие шаги:
- **Иерархический сбор ссылок (Оглавление):** Примените алгоритм для соответствующего паттерна навигации.
- **Асинхронный парсинг:** С помощью `asyncio.Semaphore` (обычно с лимитом 5 одновременных вкладок) запустить параллельное посещение страниц.
- **Блокировка лишних ресурсов:** Используйте `page.route` для блокировки загрузки изображений и аналитики. Это ускоряет работу в десятки раз и спасает от таймаутов (`page.route("**/*.{png,jpg,jpeg,webp,gif,svg}", lambda r: r.abort())`).
- **Очистка HTML перед парсингом:** Удаляйте кнопки "Скопировать" внутри `<pre>` и извлекайте чистый `innerText`, чтобы избавиться от тегов подсветки синтаксиса. Удаляйте хлебные крошки (`breadcrumbs`).
- **Извлечение и конвертация:** Сконвертируйте HTML в Markdown через `html2text` с параметром `body_width = 0`. Добавьте в начало каждого текста HTML-якорь (`<a id="page-X"></a>`).
- **Сохранение и объединение:** Сгенерируйте интерактивное **иерархическое Оглавление

### Проблема: Интерактивные табы (вкладки кода) стираются React
- **Кейс:** Next.js (App Router / React 19)
- **Симптомы:** На странице есть блоки с вкладками (например, выбор `npm`, `pnpm`, `yarn`). Если мы пытаемся прокликать вкладки, сгенерировать собственный HTML с их содержимым и использовать `parentBlock.replaceWith(newBlock)`, то React мгновенно замечает рассинхрон виртуального DOM с реальным и откатывает наши изменения. В итоге возвращается пустой или задублированный блок.
- **Решение:** Нельзя использовать замену существующих узлов (`replaceWith`), так как React откатывает изменения. Кроме того, в Next.js клик по табу (например, `npm`) переключает состояние **глобально по всей странице**. 
  
  **Рабочий алгоритм:**
  1. Сначала собираем список всех уникальных названий табов на странице (со всех блоков кода).
  2. В цикле кликаем по каждому уникальному имени таба с помощью **встроенного метода Playwright** (`page.get_by_role("tab", name=...).click()`), а не через JS `evaluate`, так как JS-клик может игнорироваться синтетическими событиями React.
  3. Делаем паузу `wait_for_timeout` для отработки React Hydration.
  4. Собираем тексты всех блоков кода с активными на данный момент табами и сохраняем их в память скрипта (в Python словарь).
  5. В самом конце, одним JS-скриптом генерируем блоки с метками `MAGICSTART Имя таба MAGICEND` и вставляем их в DOM перед оригинальными блоками (с помощью `insertBefore`).
  6. На этапе конвертации (после `html2text`) заменяем метки `MAGIC` на Markdown заголовки `### [Имя таба]`. Это гарантированно сохраняет форматирование и предотвращает потерю данных.

## Описание текущих парсеров

### 1. OpenCode
- **Модуль:** `opencode`
- **Запуск:** `uv run python main.py run opencode`
- **Паттерн меню:** Вложенные `<ul>` / `<li>`

### 2. Google Antigravity
- **Модуль:** `antigravity`
- **Запуск:** `uv run python main.py run antigravity`
- **Паттерн меню:** Плоский список с вложенностью через CSS (`padding-left`).

### 3. Payload CMS
- **Модуль:** `payload`
- **Запуск:** `uv run python main.py run payload`
- **Паттерн меню:** Динамические аккордеоны. Использовано предварительное "прокликивание" кнопок в DOM перед сбором ссылок. Также применена блокировка изображений и скриптов аналитики для ускорения.

### 4. Next.js
- **Модуль:** `nextjs`
- **Запуск:** `uv run python main.py run nextjs`
- **Паттерн меню:** Вложенные `<ul>` / `<li>` (как в OpenCode, но навигация запрятана).
- **Особенности:** Документация содержит интерактивные блоки кода (табы pnpm/npm, JS/TS). В Next.js табы синхронизируются глобально (переключение языка в одном блоке переключает язык на всей странице).
  - Реализован паттерн "Сбор в память + Финальная инъекция", обходящий защиту React Virtual DOM.
  - Используются "грязные" метки (MAGICSTART/MAGICEND) для безопасного проноса названий табов через конвертер `html2text`.

### 5. Pencil
- **Модуль:** `pencil`
- **Запуск:** `uv run python main.py run pencil`
- **Паттерн меню:** Nextra Sidebar с дублированной мобильной/десктопной навигацией.
- **Особенности:** У групп в сайдбаре есть `button[data-href]`, но эти `data-href` не являются полноценными страницами документации, поэтому парсер берет только реальные `<a>`-ссылки. Основной контент извлекается из `article > main[data-pagefind-body]`, а для блоков кода удаляются кнопки и сохраняется чистый текст из `pre`.
