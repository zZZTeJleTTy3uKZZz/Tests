# Product Requirements Document (PRD) — База знаний технической документации

## 1. Введение

Модуль «База знаний технической документации» представляет собой Backend-систему для централизованного учета, управления и взаимодействия со спарсенными техническими документациями. Платформа будет служить мостом между генераторами документации (парсерами) и ИИ-агентами (через RAG-системы).

## 2. Цели проекта

- Ведение строгого учета всех парсеров и собранных документов.
- Разделение документаций на фрагментированные базы знаний (1 инструмент = 1 блокнот NotebookLM) с сохранением ID блокнотов.
- Обеспечение API для подключения ИИ-агентов к локальным и внешним RAG (через NotebookLM MCP сервер).
- Подготовка фундамента для общего RAG (например, LightRAG), позволяющего делать кросс-запросы ко всей базе знаний одновременно.

## 3. Спецификации сущностей (Схема БД)

Главная сущность: **Документация (DocumentationEntity)**

| Поле            | Тип         | Описание                                                                                                           |
| --------------- | ----------- | ------------------------------------------------------------------------------------------------------------------ |
| `id`            | UUID/Int    | Уникальный идентификатор записи                                                                                    |
| `name`          | String      | Название инструмента (напр., FastAPI, React, Payload CMS)                                                          |
| `status`        | Enum        | Текущий статус: _Планируется, Создание парсера, Парсится, Загружается в RAG, Актуальная версия, Устаревшая версия_ |
| `version`       | String      | Версия документации (напр., "1.0.0", "latest")                                                                     |
| `direction`     | Enum/String | Направление: _ИИ агенты, Разработка, Маркетинг, DevOps и т.д._                                                     |
| `category`      | Enum/String | Категория: _Язык программирования, Сервис, Приложение, Регламент, Фреймворк и т.д._                                |
| `doc_type`      | Enum/String | Тип: _Техническая документация, Инструкции, Курсы, YouTube каналы и т.д._                                          |
| `notebooklm_id` | String      | ID блокнота в NotebookLM (опционально, если статус не дошел до "Загружается в RAG")                                |
| `source_url`    | String      | Ссылка на оригинальный источник                                                                                    |
| `local_path`    | String      | Путь к сгенерированному markdown файлу                                                                             |
| `created_at`    | DateTime    | Дата создания записи                                                                                               |
| `updated_at`    | DateTime    | Дата последнего обновления                                                                                         |

## 4. Архитектура RAG

Система будет использовать двухуровневый подход:

1. **Фрагментированный (Локальный) контекст**: Для конкретных технических вопросов по одному инструменту используется **NotebookLM**. Каждый инструмент загружается в свой блокнот. Бэкенд сохраняет `notebooklm_id`. Взаимодействие ИИ агента происходит с помощью:
   - _NotebookLM MCP сервер_ (`epicaltrendweb/notebooklm-mcp-v2`). Запущен как соседний Docker-контейнер и отдает HTTP REST API. Бэкенд на FastAPI общается с ним для создания новых блокнотов, загрузки и автоматической фрагментации больших Markdown-документов (чтобы обходить лимиты Google в 200МБ или 500k слов на источник).
   - _NotebookLM Skill_ (`PleasePrompto/notebooklm-skill`) для прямого обращения агента в нужный контекст из CLI.
2. **Глобальный контекст (Общее окно)**: Для вопросов "Сравни FastAPI и Express" или запросов ко всей базе. Реализация через поднятие локального RAG (например, **LightRAG**), куда параллельно с NotebookLM загружаются все документы с метаданными. Graph-mode отлично подойдет для связывания смежных технологий.

## 5. План реализации

**Этап 1: Инициализация и БД (✅ Выполнено)**

1. Инициализировать проект с помощью `uv` (`uv init`).
2. Установить зависимости: FastAPI, SQLAlchemy, Alembic, Pydantic, Uvicorn, asyncpg (или sqlite для старта).
3. Создать модели SQLAlchemy в соответствии со спецификацией.
4. Настроить Alembic и создать первую миграцию.

**Этап 2: Инфраструктура и Контейнеризация (✅ Выполнено)**

1. Объединить FastAPI (Backend) и NotebookLM MCP сервер в единый `docker-compose.yml`.
2. Настроить проброс локальной папки спарсенных документов (`docs_parsing`) в оба контейнера по единому пути `/app/docs`.
3. Настроить проброс системного прокси (Hiddify) в контейнеры для обхода блокировок.
4. Убедиться в персистентности сессий (проброс папки `deploy-package/data` для MCP-сервера).

**Этап 3: CRUD API и Внутренняя логика (🔄 В процессе)**

1. Разработать эндпоинты для создания, чтения, обновления и удаления записей документации (`/api/v1/docs`).
2. Добавить фильтрацию (по статусу, категории, направлению).
3. Протестировать внутреннюю бизнес-логику умной обрезки Markdown-файлов (`split_markdown_file`).

**Этап 4: Интеграция с NotebookLM MCP (🔄 В процессе)**

1. Протестировать эндпоинт/скрипт (`process_document_for_rag`), который:
   - Читает разбитый Markdown-файл.
   - Создает новый блокнот в NotebookLM.
   - Загружает все части файла в этот блокнот.
   - Возвращает и сохраняет `notebooklm_id` в базу данных.
2. Синхронизировать API с сервером NotebookLM (скрейпинг текущих блокнотов и актуализация их в БД).
3. Разработать роутер-посредник (`/ask`), который будет маршрутизировать запросы ИИ-агентов в нужный блокнот по ID.

**Этап 5: Отложенные задачи (В бэклоге)**

1. _Общий RAG (LightRAG)_: Развертывание графового RAG для кросс-инструментальных запросов.
2. _Миграция на PostgreSQL_: Перевод базы данных с SQLite на PostgreSQL при увеличении нагрузки.
