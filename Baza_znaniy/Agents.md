# Инструкция для ИИ-Агентов (База знаний технической документации)

Данный модуль представляет собой Backend-систему (на базе FastAPI) для централизованного учета, управления и взаимодействия со спарсенными техническими документациями. Платформа служит мостом между генераторами документации (парсерами) и ИИ-агентами (через RAG-системы).

> **🔥 ОБЯЗАТЕЛЬНОЕ ПРАВИЛО:** При любых изменениях архитектуры, добавлении новых интеграций (особенно MCP серверов или новых RAG-механизмов) или изменениях структуры БД вы **ОБЯЗАНЫ** актуализировать этот файл и при необходимости дополнять PRD в папке `docs/`.

## 🏗 Архитектура проекта и Структура директорий

Проект разделен на две основные части:
- `docs/` — содержит документацию, PRD (описание требований, сущностей и архитектуры), и README.
- `src/` — содержит весь исходный код приложения.

### Внутри `src/`:
- **`app/`** — Основной код FastAPI приложения:
  - `api/` — Роутеры и эндпоинты (напр. CRUD для документации).
  - `core/` — Базовые настройки (если есть) и конфигурация.
  - `db/` — Настройки подключения к БД (SQLite/PostgreSQL) и сессии (через SQLAlchemy + async).
  - `models/` — SQLAlchemy модели таблиц базы данных (`knowledge_base.py`).
  - `schemas/` — Pydantic схемы для валидации данных, входящих и исходящих запросов.
  - `services/` — Бизнес-логика и фоновые задачи (напр., интеграция с RAG: `rag_service.py`).
- **`migrations/` и `alembic.ini`** — Инфраструктура миграций БД Alembic.
- **`main.py`** — Точка входа в FastAPI приложение.
- **`Dockerfile` и `docker-compose.yml`** — Настройки контейнеризации приложения.
- **`pyproject.toml`** — Управление зависимостями через `uv`.

## 🛠 Используемые навыки и инструменты

- **Менеджер пакетов:** `uv`. Всегда используйте команду `uv add <package>` для установки пакетов и `uv run` для запуска скриптов.
- **Фреймворк:** FastAPI + Uvicorn.
- **База Данных:** Асинхронная SQLAlchemy. Изначально используется SQLite (`aiosqlite`), но заложен фундамент (и зависимости `asyncpg`, `psycopg2`) для PostgreSQL.
- **Миграции:** Alembic.
  - Создание миграции: `uv run alembic revision --autogenerate -m "Название"`
  - Применение миграций: `uv run alembic upgrade head`

> **ВНИМАНИЕ:** При работе с командами `uv`, `alembic` или `pytest` всегда проверяйте, что находитесь в директории `src/` (или используйте параметр `workdir: ".../src"` в вызове bash).

## 🧠 Интеграция с RAG (NotebookLM & LightRAG)

Стратегия платформы предполагает двухуровневый RAG:
1. **Фрагментированный (Локальный) контекст (NotebookLM):** 
   - Каждый инструмент получает свой блокнот. 
   - Загрузка файлов инициируется через `app.services.rag_service` (фоновые задачи FastAPI).
   - *Для агентов:* В будущем здесь должен быть реализован MCP-клиент, который стучится в сервер `epicaltrendweb/notebooklm-mcp-v2` для загрузки `.md` файлов и сохраняет `notebooklm_id` в БД.
2. **Глобальный контекст (LightRAG):**
   - Единое окно для кросс-инструментальных запросов.

## 🚀 Правила разработки для Агентов

1. **Запуск и проверка:** Перед завершением задачи убедитесь, что приложение запускается (в `src/` выполните `uv run uvicorn main:app --host 0.0.0.0 --port 8000`). Если задача связана с Docker — проверьте `docker-compose up -d.
2. **Работа с путями:** Локальный путь к сгенерированному markdown (`local_path`) в БД ссылается на результаты из соседнего модуля (`../Парсинги документаций/`). Учитывайте это при монтировании volume-ов в Docker (`docker-compose.yml` уже пробрасывает эту папку в `/app/docs`).
3. **Обновление БД:** Если вы добавили поле в модель `app/models/knowledge_base.py` — **ОБЯЗАТЕЛЬНО** сделайте миграцию через alembic:
   ```bash
   cd src/
   uv run alembic revision --autogenerate -m "Add new field"
   uv run alembic upgrade head
   ```
4. **Схемы и API:** Любые изменения в моделях должны синхронно отражаться в Pydantic-схемах (`app/schemas/`) и, если нужно, в роутерах (`app/api/`).
5. **Фоновые задачи:** При интеграции длительных операций (например, загрузка больших Markdown в RAG), обязательно используйте `BackgroundTasks` FastAPI, чтобы не блокировать ответ API.
