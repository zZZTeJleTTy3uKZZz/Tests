# ADR-001: Стратегия интеграции с NotebookLM

## Статус

Предлагается

## Контекст

Текущий репозиторий уже интегрирован с `NotebookLM` через vendored форк `notebooklm-mcp`, встроенный в Python-проект. Это дало быстрый старт, но породило несколько проблем:

- Python-продукт жёстко зависит от отдельного TypeScript-сервиса;
- внешний слой уже завязан на конкретные HTTP-эндпоинты и файловые пути контейнера;
- переаутентификация Google/NotebookLM остаётся хрупкой;
- для целевого продукта нужны ingestion, freshness monitoring и multi-notebook query orchestration, а не просто прокси к одному backend;
- пользователь хочет строить систему на актуальной документации и современных практиках, а также иметь управляемый CI-контур.

Дополнительно локально подтверждено, что CLI `notebooklm-py` версии `0.3.4` предоставляет нужные базовые операции:

- `login`
- `auth check`
- `list`
- `create`
- `ask`
- `source add`
- `source stale`
- `source refresh`
- `metadata`

Также подтверждено, что в `notebooklm-py` логин завершается сохранением auth-state после нажатия `ENTER` в терминале, что совпадает с наблюдаемым пользовательским сценарием ручного восстановления сессии.

## Драйверы решения

- нужен Python-first стек для orchestration-слоя;
- нужен управляемый и тестируемый adapter layer;
- нужен путь к надёжной проверке auth-state;
- нужен CI-friendly контур probe/sync/reporting;
- нужна возможность временно не потерять уже реализованный функционал `MCP`.

## Рассмотренные варианты

## Вариант 1. Оставить `notebooklm-mcp` единственным backend

Плюсы:

- уже встроен в текущий репозиторий;
- там есть готовые HTTP-эндпоинты;
- уже реализованы session manager, library и re-auth механики.

Минусы:

- высокий уровень связанности;
- сложнее сопровождать в Python-продукте;
- тяжело эволюционировать ingestion и sync вокруг чужого большого форка;
- проблемы auth не исчезают автоматически.

## Вариант 2. Полностью выкинуть `MCP` и сразу перейти только на `notebooklm-py`

Плюсы:

- единый Python-стек;
- проще строить адаптер, worker и API;
- понятнее отладка и развёртывание.

Минусы:

- можно быстро потерять часть уже доступного функционала;
- миграция может занять больше времени;
- потребуется самостоятельно добрать некоторые edge-case сценарии.

## Вариант 3. Ввести адаптерный слой и сделать гибридную стратегию

Плюсы:

- можно использовать `notebooklm-py` как основной путь;
- можно оставить `MCP` как fallback и источник зрелых workaround-механик;
- доменная логика перестаёт зависеть от конкретного backend;
- переход можно делать инкрементально, без большого взрыва.

Минусы:

- временно придётся поддерживать два backend-адаптера;
- потребуется аккуратное capability routing.

## Решение

Принять **гибридную стратегию**:

- ввести интерфейс `NotebookLMAdapter` внутри Python-приложения;
- сделать `NotebookLMPyAdapter` основным адаптером для нового ingestion/sync/query слоя;
- оставить `NotebookLMMcpAdapter` как временный fallback и совместимый legacy backend;
- убрать прямую бизнес-зависимость роутов и сервисов от `MCP`-эндпоинтов;
- поднять auth-check, sync и query orchestration как first-class подсистемы в FastAPI.

## Последствия

### Положительные

- ядро продукта становится Python-first и ближе к основной кодовой базе;
- можно эволюционировать бизнес-логику без жёсткой привязки к одному backend;
- проще строить health-check, refresh workflow и CI-задачи;
- можно мигрировать постепенно и не ломать уже имеющийся функционал.

### Отрицательные

- некоторое время придётся сопровождать два способа интеграции;
- потребуется слой capability detection;
- нужны дополнительные тесты на совместимость двух адаптеров.

### Риски

- у `notebooklm-py` тоже браузерная auth-модель, это не настоящий серверный OAuth;
- некоторые сценарии всё равно будут требовать ручного refresh auth-state;
- feature parity между `MCP` и `notebooklm-py` может оказаться неполной.

## Смягчение рисков

- auth-состояние вынести в отдельный health workflow;
- запросы и sync-задачи не выполнять, если auth-state невалиден;
- держать fallback на `MCP` до достижения нужной функциональной полноты;
- для CI разделить задачи probe/reporting и задачи, требующие живой браузерной сессии.

## Практическое следствие для реализации

Порядок действий:

1. Ввести `NotebookLMAdapter`.
2. Реализовать `NotebookLMPyAdapter`.
3. Перенести создание блокнота, добавление источника и `ask` на новый слой.
4. Сохранить `MCP`-fallback через конфиг.
5. Добавить auth-check и refresh orchestration.
