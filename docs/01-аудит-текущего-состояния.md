## Короткий вывод

В репозитории уже есть хороший фундамент, но не готовый продукт.

Готово:

- прототипы парсеров документации;
- FastAPI-сервис базы знаний с CRUD по документам;

## Где сейчас главный технический долг

## 1. Нет единого ingestion-пайплайна

Сейчас есть куски:

- парсинг;
- запись в БД;
- загрузка в `NotebookLM`.

Но нет одной транзакционной цепочки с явными состояниями и повторными попытками.

## 2. Нет нормального refresh-цикла

Продукту нужна модель:

- отследить, что источник устарел;
- понять, изменился ли changelog или сам контент;
- перепарсить только нужные разделы;
- обновить источники в `NotebookLM`;
- зафиксировать новый хеш и дату синхронизации.

Сейчас этого слоя нет.

## 3. Нет batch-query orchestration

Текущий `/ask` заточен под один блокнот. Пользовательская задача требует:

- пробросить запрос сразу в несколько релевантных блокнотов;
- дождаться всех результатов;
- нормализовать ответы;
- вернуть единый JSON;
- по возможности приложить источники и confidence.

## 4. Слабая модель статусов документа

Текущие статусы полезны только для ручного наблюдения. Для продукта нужны отдельные фазы:

- обнаружение структуры;
- парсинг;
- нормализация;
- загрузка;
- актуально;
- ожидает обновления;
- обновляется;
- ошибка синхронизации;
- ошибка авторизации;
- требует ручного вмешательства.

## 5. Аутентификация `NotebookLM` нестабильна не по одной причине

По коду видно сразу несколько факторов риска:

- серверная сессия Google может истечь раньше, чем локально выглядят валидными cookies;
- состояние авторизации и `sessionStorage` живут отдельно;
- в compose не настроен явный устойчивый сценарий авто-логина;
- FastAPI-внешний слой не управляет переаутентификацией как first-class workflow;
- состояние сессии завязано на браузерный профиль и файловые артефакты.

## Что особенно важно по `notebooklm-py`

Локально подтверждено через CLI `notebooklm-py` версии `0.3.4`:

- есть команды `login`, `list`, `create`, `use`, `ask`;
- есть `source add`, `source stale`, `source refresh`, `metadata`;
- есть `auth check`;
- команда `login` явно говорит: после логина нужно нажать `ENTER` в терминале, чтобы сохранить аутентификацию.

Это очень хорошо совпадает с твоим наблюдением:

- “открывается профиль, я просто жму Enter, и новая сессия сохраняется”.

Вывод:

- как минимум часть боли не баг уровня продукта, а ожидаемая модель сохранения auth-state в текущем CLI;
- следовательно, поверх `notebooklm-py` можно построить контролируемый Python-адаптер с явным health-check и сценариями refresh/relogin.

## Итог аудита

Сейчас у тебя не пустой проект, а сырой, но уже ценный фундамент.

Самое сильное, что уже есть:

- практический опыт сложного парсинга документации;
- рабочий Python API-слой;
- продвинутый `NotebookLM`-форк как источник идей и fallback-механик.

Самое слабое место:

- нет чистой модульной архитектуры вокруг ingestion, sync и query orchestration;
- `NotebookLM`-интеграция пока слишком хрупкая и слишком завязана на конкретный `MCP`-форк;
- CI и фоновые обновления пока не оформлены как отдельные управляемые процессы.
