# Roadmap и бэклог реализации

## Цель релиза MVP

Собрать систему, которая:

- принимает URL документации;
- парсит её в структурированный Markdown с оглавлением;
- создаёт под неё блокнот в `NotebookLM`;
- умеет хранить статус и актуальность базы знаний;
- умеет выполнять запросы по одному или нескольким блокнотам;
- умеет отслеживать устаревание источников.

## Фаза 0. Техническая расчистка

### Story 1. Привести проект к модульной структуре

**Как** разработчик  
**Я хочу** выделить ingestion, sync и query в отдельные пакеты  
**Чтобы** код был тестируемым и расширяемым

Критерии приёмки:

- `app/services` разделён хотя бы на `ingestion`, `sync`, `query`, `notebooklm`;
- прямые вызовы внешнего backend-слоя не размазаны по роутам;
- у `NotebookLM` есть интерфейс-адаптер.

### Story 2. Ввести расширенную модель статусов

**Как** система  
**Я хочу** хранить явные стадии lifecycle документа  
**Чтобы** можно было нормально понимать, что происходит

Критерии приёмки:

- статусы отражают `queued/parsing/uploading/ready/stale/refreshing/failed/auth_required`;
- миграция БД добавлена;
- API возвращает новые статусы.

## Фаза 1. Universal ingestion

### Story 3. Принимать URL и создавать ingestion job

**Как** пользователь  
**Я хочу** отправить ссылку на документацию  
**Чтобы** система сама начала разбор и загрузку

Критерии приёмки:

- есть `POST /api/v1/ingestions`;
- создаётся job и запись документа;
- пользователь получает `job_id`.

### Story 4. Реализовать detector стратегий парсинга

**Как** ingestion-сервис  
**Я хочу** определять тип docs-сайта и выбирать стратегию  
**Чтобы** не плодить ручные if-else в одном месте

Критерии приёмки:

- есть `parser registry`;
- для неизвестного сайта включается fallback-стратегия;
- решение и причина выбора сохраняются в job metadata.

### Story 5. Стандартизировать результат парсинга

**Как** downstream-модуль  
**Я хочу** получать единый формат парсинга  
**Чтобы** загрузка и diff не зависели от конкретного parser implementation

Критерии приёмки:

- создаются `manifest.json`, `toc.json`, `full.md`, `pages/*.md`;
- у каждой страницы есть title, slug, source_url, order и hash;
- артефакты сохраняются на диск в предсказуемой структуре.

## Фаза 2. NotebookLM adapter

### Story 6. Сделать основной Python-адаптер над `notebooklm-py`

**Как** backend  
**Я хочу** работать с `NotebookLM` через Python-обёртку  
**Чтобы** убрать лишнюю хрупкость межъязыковой интеграции

Критерии приёмки:

- есть методы `login/status/create_notebook/add_source/ask/source_stale/source_refresh/metadata`;
- все ошибки адаптера нормализованы;
- есть structured logging.

### Story 7. Сделать health-check и auth-check

**Как** система  
**Я хочу** до запуска job понимать, валиден ли auth-state  
**Чтобы** не падать посреди загрузки без контекста

Критерии приёмки:

- есть `GET /api/v1/health/notebooklm`;
- есть `POST /api/v1/notebooklm/re-auth`;
- если auth недоступен, job переводится в `auth_required`.

## Фаза 3. Knowledge base и freshness

### Story 8. Отслеживать изменения источников

**Как** владелец базы знаний  
**Я хочу** видеть, какие документы устарели  
**Чтобы** система держала актуальную документацию

Критерии приёмки:

- хранится `etag`, `last_modified`, `content_hash`, `changelog_hash`;
- есть периодическая задача probe;
- stale документы видны через API.

### Story 9. Реализовать refresh workflow

**Как** система  
**Я хочу** уметь обновлять источник без ручной пересборки  
**Чтобы** база знаний жила сама

Критерии приёмки:

- есть `POST /api/v1/documents/{id}/refresh`;
- если `NotebookLM` умеет refresh источника, он вызывается;
- если нет, запускается полный re-ingestion.

## Фаза 4. Query orchestration

### Story 10. Спросить один блокнот через стабильный API

**Как** AI-агент  
**Я хочу** задавать вопрос через наш API  
**Чтобы** не знать про внутренности `NotebookLM`

Критерии приёмки:

- есть `POST /api/v1/query`;
- в ответе есть answer, sources, latency, notebook metadata;
- ошибки возвращаются в нормальном JSON.

### Story 11. Спросить несколько блокнотов и получить единый ответ

**Как** AI-агент  
**Я хочу** задавать вопрос нескольким knowledge notebooks  
**Чтобы** получать агрегированный grounded answer

Критерии приёмки:

- есть `POST /api/v1/query/batch`;
- система ждёт все целевые запросы;
- возвращается merged answer и список индивидуальных результатов.

### Story 12. Добавить deterministic routing

**Как** оркестратор  
**Я хочу** сам выбирать релевантные блокноты без магии  
**Чтобы** ответ был воспроизводимым

Критерии приёмки:

- роутинг работает по тегам, provider, category и topic;
- логируется, почему был выбран каждый блокнот;
- доступен dry-run режим.

## Фаза 5. CI и эксплуатация

### Story 13. Сделать фоновый отчёт по свежести knowledge base

**Как** владелец продукта  
**Я хочу** ежедневно видеть, что устарело и что сломалось  
**Чтобы** не узнавать об этом руками

Критерии приёмки:

- есть scheduled job;
- отчёт содержит stale docs, failed jobs, auth issues;
- результат пишется в файл или артефакт CI.

### Story 14. Сделать smoke-набор для критических сценариев

**Как** команда разработки  
**Я хочу** быстро проверять базовый happy path  
**Чтобы** не ломать ingestion и query при доработках

Критерии приёмки:

- есть smoke test на parser;
- есть smoke test на adapter;
- есть smoke test на query API.

## Предлагаемый порядок реализации

1. Разделить доменные модули и статусы.
2. Ввести `NotebookLMAdapter`.
3. Поднять ingestion job model.
4. Перевести создание блокнотов и загрузку источников на новый адаптер.
5. Реализовать single-query API.
6. Реализовать batch-query API.
7. Добавить freshness probe и refresh workflow.
8. Добавить CI smoke и ежедневный отчёт.

## Definition of Done для первого рабочего релиза

- можно зарегистрировать новый URL документации;
- система строит артефакты и создаёт блокнот;
- документ получает статус `ready`;
- можно спросить один блокнот и несколько блокнотов;
- можно увидеть stale документы;
- при истекшей сессии система не падает молча, а возвращает понятный статус `auth_required`.
